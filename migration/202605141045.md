**Takeaway:**

# 🟥 1. What `StatusSurface` _is_

`StatusSurface` is a **UI_telemetry_surface**:  
a single‑row, high‑signal, low‑noise strip that communicates:

- current **UIMode**
- current **active host**
- current **scan state**
- ephemeral operator messages
- time / tick indicator
- system stance (NORMAL, SCANNING, DIFF, ERROR)

It is the **operator’s grounding line** — the thing that keeps the whole TUI cognitively stable.

---

# 🟦 2. What `StatusSurface` _contains_

### **a) Local ephemeral state**

It holds only what it needs to render:

- `message: Optional[str]`
- `message_ttl: int` (ticks until cleared)
- `last_tick: datetime`
- `scan_indicator: Optional[str]` (spinner, progress bar, etc.)

This state is **ephemeral**, not structural.  
It never stores host data, scan results, or inventory.

---

### **b) Render function**

The render method consumes **SurfaceProps** and draws:

- left: mode + host
- center: scan status
- right: clock + ephemeral message

This is the **operator’s HUD**.

---

### **c) No message emission**

Unlike other surfaces, `StatusSurface` does **not** emit messages.

It is **read‑only** from the operator’s perspective.

All updates come from:

- root loop (e.g., “scan started”, “scan finished”)
- props (e.g., active host changed)

This keeps it pure and predictable.

# 🟩 3. What `StatusSurface` _does NOT_ do

It does **not**:

- trigger scans
- handle input
- manage host selection
- store scan results
- own any domain logic
- perform side‑effects
- communicate with the scanner engine

It is **pure UI telemetry** — nothing more, nothing less.

This is what keeps the substrate clean.
