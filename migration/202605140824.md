# **1. Transition: Signal → Framing**

### **Core failure modes**

- **Drift:** signal noticed but not captured
- **Overload:** too many signals, no prioritization
- **Misrouting:** emotional or associative framing instead of operational framing

### **Stabilizing invariants**

- **Xi‑Invariant:** _No state change without a named intent and a timestamp._  
    → Forces the signal to become an object, not a feeling.
- **Razor‑Invariant:** _If the signal cannot be framed in ≤2 sentences, it is not a signal._  
    → Prevents runaway associative drift.
- **Guard‑Invariant:** _Framing must specify domain: operator, translator, storyteller, or witness._  
    → Prevents misrouting into the wrong cognitive mode.

---

# **2. Transition: Framing → Selection**

### **Core failure modes**

- **Drift:** endless reframing without choosing
- **Overload:** too many viable paths
- **Misrouting:** selecting based on urgency instead of mission

### **Stabilizing invariants**

- **Xi‑Invariant:** _Selection requires a declared success condition._  
    → You cannot choose without knowing what “done” looks like.
- **Razor‑Invariant:** _Kill all branches that do not serve the active mission in ≤2 steps._  
    → Cuts noise, preserves mission integrity.
- **Operator‑Invariant:** _Selection must bind to a resource: time, energy, or tool._  
    → Prevents abstract choices that never materialize.

---

# **3. Transition: Selection → Commitment**

### **Core failure modes**

- **Drift:** “I’ll do it later” with no anchor
- **Overload:** committing to too many selections
- **Misrouting:** committing emotionally instead of structurally

### **Stabilizing invariants**

- **Xi‑Invariant:** _Commitment requires a timestamp and a container._  
    → Container = where it lives (note, task, vault entry).
- **Razor‑Invariant:** _If commitment cannot survive scrutiny, collapse it immediately._  
    → Prevents false commitments that rot.
- **Guard‑Invariant:** _Commitment must specify the first physical action._  
    → Forces embodiment, not intention‑vapor.

---

# **4. Transition: Commitment → Execution**

### **Core failure modes**

- **Drift:** starting without grounding
- **Overload:** execution begins before the system is ready
- **Misrouting:** using the wrong tool or wrong cognitive mode

### **Stabilizing invariants**

- **Xi‑Invariant:** _Execution requires a stable entry point._  
    → A single, unambiguous “start here.”
- **Razor‑Invariant:** _If execution forks, kill all forks except the one that moves the mission forward._  
    → Prevents runaway improvisation.
- **Operator‑Invariant:** _Execution must be observable._  
    → If you can’t see it, you can’t steer it.

---

# **5. Transition: Execution → Review**

### **Core failure modes**

- **Drift:** never reviewing, losing lineage
- **Overload:** too much output, no compression
- **Misrouting:** reviewing emotionally instead of structurally

### **Stabilizing invariants**

- **Xi‑Invariant:** _Review must produce a compression artifact._  
    → A summary, a note, a decision — something that persists.
- **Razor‑Invariant:** _If the review reveals contradiction, resolve via Reconcile, Split, or Elevate._  
    → Prevents cognitive knots.
- **Guard‑Invariant:** _Review must update the mission state._  
    → Ensures continuity and prevents orphaned threads.
