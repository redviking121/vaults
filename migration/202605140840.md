# **Vault Stability Checklist v0.1**

**Takeaway:** If this checklist passes cleanly, the substrate is stable enough to introduce Logseq in a controlled, one‑way pipeline.

---

## **1. Obsidian Vault Stability (Engine Room)**

These checks confirm that the local vault behaves deterministically and that no plugin or sync mechanism is introducing drift.

### **1.1 Folder & File Integrity**

- Verify the vault root contains only expected top‑level directories.
- Confirm no “.obsidian” subfolder drift (duplicate configs, ghost workspace files).
- Run a recursive `stat` check to ensure no files have timestamps that “jump” unexpectedly.
- Confirm no zero‑byte files unless intentionally created.

### **1.2 Plugin Determinism**

- Disable all community plugins.
- Re‑enable them **one by one**, watching for:
    - unexpected file writes
    - auto‑generated folders
    - metadata injection
    - background indexing loops
- Confirm the plugin list matches your known, intentional loadout.

### **1.3 Namespace Hygiene**

- Search for:
    - orphaned notes
    - broken links
    - dangling embeds
    - duplicated slugs
- Confirm all internal links resolve cleanly.

### **1.4 Index Regeneration**

- Run your `chronoindex.py` job manually.
- Confirm `_index.md` regenerates without:
    - missing entries
    - duplicate entries
    - phantom pages
- Verify the index diff is clean and predictable.

### **1.5 Sync Drift Check**

- If using Obsidian Sync or Drive sync:
    - confirm no conflict files
    - confirm no “(1)” duplicates
    - confirm no timestamp churn
- Run a diff between local and cloud copies to ensure parity.

---

## **2. Ops‑Vault Stability (Bridge Layer)**

This is the layer that connects your local vault to the outside world. If this is unstable, Razor will amplify the instability.

### **2.1 rclone Mount Stability**

- Confirm the systemd user service is active and not restarting.
- Check logs for:
    - mount retries
    - rate‑limit errors
    - partial writes
- Verify the mountpoint is writable and latency is stable.

### **2.2 Directory Parity**

- Compare Ops‑Vault root with expected structure.
- Confirm no recursive sync loops (Drive ↔ local ↔ Drive).
- Ensure no hidden `.tmp` or `.sync` artifacts.

### **2.3 Timestamp Consistency**

- Spot‑check a handful of files across local and cloud.
- Confirm timestamps match within expected tolerances.
- Look for churn — repeated updates without content changes.

### **2.4 Ghost File Sweep**

- Search for:
    - files with no extension
    - files with random hashes
    - partial uploads
- Remove or quarantine anything suspicious.

---

## **3. Environment Stability (Operator Machine)**

This ensures the environment itself isn’t introducing nondeterminism.

### **3.1 Python Environment**

- Confirm Python 3.12 venv for Open‑WebUI is stable.
- Verify no dependency drift (`pip freeze` diff).
- Ensure no global Python packages are shadowing venv packages.

### **3.2 Filesystem Health**

- Run a quick filesystem check for corruption.
- Confirm no inode exhaustion.
- Ensure no background processes are touching the vault unexpectedly.

### **3.3 GNOME / Ubuntu Noble Stability**

- Confirm no recent crashes or extension failures.
- Ensure no auto‑backup or indexing services are scanning the vault.

---

## **4. Lineage & Identity Checks (Substrate Sanity)**

This ensures the vault’s conceptual structure hasn’t drifted.

### **4.1 Namespace Lineage**

- Confirm all top‑level namespaces still map to their intended conceptual domains.
- Verify no “leakage” — e.g., research notes appearing in operational folders.

### **4.2 Operator Notes Integrity**

- Check HEADLINE drills, Axioms, and Operator Stance notes for accidental edits.
- Confirm no accidental overwrites or merges.

### **4.3 Razor Substrate Consistency**

- Ensure the Xi/Razor notes haven’t forked into multiple versions.
- Confirm the Decision Stability Rules (DSR) notes are intact and unchanged.

---

## **5. Final Determinism Test (The Gatekeeper)**

This is the last check before Logseq enters the pipeline.

### **5.1 Controlled Write Test**

- Create a test note.
- Sync it.
- Delete it.
- Confirm the deletion propagates cleanly with no ghosts.

### **5.2 Controlled Rename Test**

- Rename a note.
- Confirm link updates behave as expected.
- Confirm no duplicate files appear.

### **5.3 Controlled Metadata Test**

- Add frontmatter to a test note.
- Confirm Obsidian doesn’t rewrite or reorder it unexpectedly.

If all three pass, the substrate is stable.

---

# **Outcome**

If this checklist passes cleanly, you have a **stable substrate** and can safely introduce Logseq in a **sandboxed, one‑way pipeline** without risking contamination of the real vault.

If anything fails, we isolate the failure and fix it before proceeding.
