# Migration Checklist ◉

Use this protocol when Vault A is being restructured, refactored, reorganized, or migrated.  
The goal is to maintain continuity of command without introducing drift or multiplying mysteries.

---

## 1. Pre‑Migration Prep
- [ ] Confirm Ops Vault is operational.
- [ ] Switch daily operations to Ops Vault.
- [ ] Freeze deep edits in Vault A.
- [ ] Backup Vault A (optional but wise).
- [ ] Note current project entrypoints:
  - [[project-entrypoints]]

---

## 2. During Migration
- [ ] Capture all new tasks in Ops Vault.
- [ ] Avoid modifying Vault A until stable.
- [ ] Keep a running list of items that need reintegration.
- [ ] Use dashboards for continuity:
  - [[cockpit]]
  - [[today]]

---

## 3. Post‑Migration Reintegration
- [ ] Reopen Vault A once stable.
- [ ] Migrate cards back into canonical locations.
- [ ] Recreate or merge notes as needed.
- [ ] Reconnect project entrypoints.
- [ ] Verify folder structure and links.

---

## 4. Final Checks
- [ ] Remove temporary notes.
- [ ] Confirm no drift occurred.
- [ ] Resume normal operations in Vault A.
- [ ] Ops Vault remains untouched and ready for next time.

---

## Operator Reminder
Migration is surgery.  
Ops Vault is the recovery room.  
Do not mix the two.
