**(Logseq → Obsidian → Ops‑Vault)**

---

# **0. PURPOSE**

To ensure that:

- Logseq blocks become Obsidian notes **without drift**
- structure is preserved
- context is preserved
- identity is preserved
- versioning is preserved
- and nothing unstable contaminates the vault

This is the **semantic refinery → publication** handoff.

---

# **1. EXPORT TRIGGER CONDITIONS**

A block tree becomes eligible for export when:

1. **Atomicity invariant** holds
2. **Parent–child clarity** holds
3. **Identity binding** is correct
4. **Context anchoring** is complete
5. **Non‑duplication** is resolved
6. **Direction of abstraction** is sane
7. **Operator override** is not blocking export

If any fail → the block remains in Logseq staging.

---

# **2. EXPORT CLASSIFICATION**

Before export, Logseq classifies the block tree into one of six export types:

1. **Agent Identity Package**
2. **Protocol**
3. **Document**
4. **Draft**
5. **Artifact**
6. **Operator Directive**

Each type has its own export rules.

---

# **3. TRANSFORMATION STAGES**

The export pipeline has **five stages**:

---

## **Stage 1 — Block Tree Extraction (Logseq)**

Logseq extracts:

- the root block
- all child blocks
- nested structure
- tags
- references
- metadata

This becomes a **Block Tree Object (BTO)**.

**Invariant:**  
No flattening. No reordering. No semantic mutation.

---

## **Stage 2 — Semantic Flattening (Logseq)**

The BTO is transformed into a **Markdown‑ready structure**:

- each block becomes a bullet
- indentation preserved
- tags preserved
- references preserved
- identity bindings preserved
- metadata becomes YAML frontmatter

This is the **semantic flattening** stage.

---

## **Stage 3 — Note Assembly (Obsidian)**

Obsidian receives the flattened structure and assembles it into a note.

### **YAML frontmatter includes:**

- `type:` (agent, protocol, document, draft, artifact, directive)
- `version:` (if applicable)
- `source:` logseq
- `created:` timestamp
- `updated:` timestamp
- `identity:` (if agent)
- `routing:` (original path)

### **Body includes:**

- block tree
- backlinks
- references
- tags
- operator notes (if any)

This is the **publication‑ready note**.

---

## **Stage 4 — Vault Routing (Obsidian → Ops‑Vault)**

Obsidian routes the note to the correct vault location:

- **Agents →** `Ops-Vault/agents/`
- **Protocols →** `Ops-Vault/governance/`
- **Documents →** `Ops-Vault/documents/`
- **Drafts →** `Ops-Vault/drafts/`
- **Artifacts →** `Ops-Vault/artifacts/`
- **Directives →** `Ops-Vault/directives/`

This is the **vault‑grade storage** stage.

---

## **Stage 5 — Operator Triage (Ops‑Vault)**

You perform:

- tagging
- linking
- versioning
- promotion
- archiving
- deletion

This is the **final human pass**.

---

# **4. SPECIAL CASES**

### **4.1 Identity Packages**

Identity packages receive:

- strict versioning
- canonical filenames
- cross‑agent references
- mesh integration tags

### **4.2 Protocols**

Protocols are:

- version‑locked
- immutable except by operator override
- cross‑linked to routing and invariants

### **4.3 Drafts**

Drafts are allowed into Obsidian **only if explicitly marked**:

```
status: draft
```

### **4.4 Artifacts**

Artifacts bypass Logseq entirely and go straight to Obsidian.

---

# **5. FAILURE MODES**

### **FM‑1: Structural Collapse**

Block tree loses hierarchy.  
→ Rebuild in Logseq.

### **FM‑2: Identity Drift**

Agent identity mutated during export.  
→ Rebind identity tags.

### **FM‑3: Context Loss**

Backlinks or references missing.  
→ Re‑anchor in Logseq.

### **FM‑4: Duplicate Export**

Same block tree exported twice.  
→ Merge or version.

---

# **6. OPERATOR OVERRIDES**

You may issue:

- “Force export.”
- “Bypass Logseq.”
- “Export as artifact.”
- “Freeze version.”

Operator overrides supersede all invariants except structural collapse.
