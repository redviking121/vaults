# **Ops‑04: Run Architecture v0.1**

_The structural backbone of every operator run._

---

## **1. Purpose and Scope**

**Purpose:**  
Define the architecture of a run so that every execution — daily, weekly, or mission‑specific — follows a predictable, inspectable, and improvable pattern.

**Scope:**  
This module governs:

- Run phases
- Run gates
- Operator stance transitions
- Artifact flow
- Failure‑mode handling
- Completion and closure

This is the doctrine that ensures a run is not “a session” but **an operation**.

---

## **2. The Run as a Structural Object**

A run is not a block of time.  
A run is a **container with invariants**.

A run has:

- **A beginning condition** (Operator Ready + Substrate Stable)
- **A middle condition** (Governance + Demonstration)
- **An end condition** (Closure + Propagation)

A run is successful when:

- The substrate is more stable than before
- The operator is more aligned than before
- The vault is richer than before
- The system is more demonstrable than before

If none of these increase, the run did not run.

---

## **3. The Three Phases (Structural)**

### **Phase 1 — STABILIZE**

_Lock the substrate._

**Objective:**  
Ensure the environment, operator, and vault are in a known‑good state.

**Actions:**

- Verify cockpit currency
- Confirm vault access
- Check environment readiness
- Identify carry‑forward items
- Establish today’s constraints

**Output Artifact:**  
**Phase 1 Integrity Snapshot**

**Success Condition:**  
No unknowns remain in the substrate.

---

### **Phase 2 — INTEGRATE**

_Bring it under governance._

**Objective:**  
Take the work that exists (raw, messy, partial, emergent) and bring it into the Xi/Razor governance structure.

**Actions:**

- Identify integration targets
- Apply Razor (cut, compress, clarify)
- Convert raw material into governed artifacts
- Update routing tables, presets, or macros
- Resolve contradictions

**Output Artifact:**  
**Phase 2 Integration Decision Record**

**Success Condition:**  
The system is more coherent than before.

---

### **Phase 3 — ACCELERATE**

_Demonstrate and propagate._

**Objective:**  
Turn the stabilized, governed system into something that _moves_.

**Actions:**

- Build demonstrations
- Publish artifacts
- Run simulations
- Execute macros
- Perform operator drills

**Output Artifact:**  
**Phase 3 After‑Action Report**

**Success Condition:**  
The system is more demonstrable than before.

---

## **4. The Gates**

Each phase has a **Gate** — a structural checkpoint that prevents drift.

### **Gate 1 → 2: Stability Gate**

You cannot integrate chaos.  
You must stabilize first.

### **Gate 2 → 3: Governance Gate**

You cannot accelerate what is not governed.  
You must integrate first.

### **Gate 3 → Close: Demonstration Gate**

You cannot close a run without demonstrating something.  
You must accelerate first.

These gates prevent the operator from skipping ahead, improvising, or collapsing phases.

---

## **5. Operator Stance Transitions**

Each phase requires a different stance.

### **Phase 1 Stance: Technician**

- Calm
- Diagnostic
- No ego
- No narrative
- Just facts

### **Phase 2 Stance: Architect**

- Structural
- Decisive
- Pattern‑seeking
- Governance‑minded

### **Phase 3 Stance: Demonstrator**

- Expressive
- Clear
- Confident
- Narrative‑aware

A run fails when the operator uses the wrong stance in the wrong phase.

---

## **6. Artifact Flow**

Every run produces:

- **1 Integrity Snapshot**
- **1 Integration Decision Record**
- **1 After‑Action Report**

Optional artifacts:

- Razor Application Record
- Sandbox Verification Log
- Demonstration Declaration Register

All artifacts flow into:

**Ops‑Vault/incoming/**  
→ triaged during Operator Cycle.

This ensures the vault grows in a controlled, governed way.

---

## **7. Failure Modes and Recovery**

### **Failure Mode 1 — Phase Collapse**

Symptoms:

- Doing Phase 2 work in Phase 1
- Doing Phase 3 work in Phase 2
- Skipping gates

Recovery:

- Stop
- Rewind to the last completed gate
- Re‑enter the correct stance

---

### **Failure Mode 2 — Operator Drift**

Symptoms:

- Emotional contamination
- Narrative takeover
- Over‑compression
- Over‑expansion

Recovery:

- HEADLINE Drill
- Re‑establish stance
- Re‑enter the phase

---

### **Failure Mode 3 — Substrate Instability**

Symptoms:

- Environment issues
- Vault access problems
- Toolchain failures

Recovery:

- Abort run
- Enter Stabilize‑Only Mode
- Produce Incident & Rollback Report

---

## **8. Completion and Closure**

A run is complete when:

- All three artifacts exist
- All three gates were passed
- The operator can state the demonstration
- The vault has been enriched
- The substrate is stable

Closure ritual:

- Mark run ID
- Publish artifacts
- Reset cockpit
- Declare readiness for next run

This closes the loop and prepares the system for the next cycle.

---

## **9. Doctrine Summary (Operator‑Grade)**

A run is a structural object with three phases, three gates, three artifacts, and three operator stances.  
It is not a session.  
It is not a block of time.  
It is a **governed operation**.

When executed correctly, a run:

- stabilizes the substrate
- integrates the system
- accelerates the operator
- enriches the vault
- increases demonstrability

This is the architecture that makes your entire ecosystem _move_.
