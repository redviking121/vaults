Xanadu Prime — Lifecycle Rules (Stub v0.1)
1. Purpose
The Lifecycle Rules define the universal temporal mechanics for all surfaces in the Prime epoch.
They ensure that every artifact:

enters the system intentionally

stabilizes deterministically

propagates safely

evolves without breaking lineage

retires explicitly

These rules prevent drift, collapse, and silent mutation.

2. Scope
The lifecycle applies to all surface classes, including:

Constitutional Surfaces

Structural Surfaces

Public Surfaces

Provenance Surfaces

Operational Surfaces

If a file exists in the repo, it is governed by these rules.

3. Lifecycle Phases
Every surface moves through three mandatory phases:

3.1 Phase 1 — Declared
The surface is named, scoped, and placed in its directory.
No content is binding yet.

A valid declaration includes:

name

class

directory

scope statement

Deepen via Naming Conventions.

3.2 Phase 2 — Stabilized
The surface is locked under DSR‑1.
Its boundaries, intent, and structural commitments are fixed.

Stabilization requires:

contradiction check

naming compliance

directory compliance

lineage registration

Razor pass

Deepen via DSR‑1.

3.3 Phase 3 — Propagated
The surface becomes part of the active corpus.
It is now authoritative and participates in the system’s structure.

Propagation includes:

integration into indices

update of structural views

provenance logging

Deepen via Provenance Timeline.

4. Optional Phase — Retired
A surface may be retired only through a constitutional event.

Retirement requires:

explicit dissolution

lineage preservation

replacement or nullification

Razor elevation

Retirement is rare and must be operator‑declared.

5. Lifecycle Invariants
These invariants apply to all surfaces:

No implicit creation — nothing enters the system without declaration.

No skipping phases — Declared → Stabilized → Propagated.

No silent mutation — stabilized surfaces cannot change without DSR‑1.

No retroactive movement — directory placement is permanent after stabilization.

Lineage continuity — every change must preserve ancestry.

Operator primacy — only the operator may advance lifecycle phases.

These invariants ensure temporal coherence.

6. Lifecycle Events
A lifecycle event is any operator action that moves a surface between phases.

Events include:

Declaration Event — creates the surface

Stabilization Event — locks the surface

Propagation Event — activates the surface

Retirement Event — dissolves the surface

Each event must be logged in the Provenance Timeline.

7. Relationship to Substrate Lifecycle
The Substrate Lifecycle governs substrate entities.
The Surface Lifecycle (this document) governs all surfaces.

They interact through:

naming

lineage

Razor

DSR‑1

Deepen via Substrate Lifecycle.

8. Out of Scope
Lifecycle Rules do not define:

runtime behavior

operator workflows

implementation details

public‑facing semantics

Those belong to other surfaces.
