Xanadu Prime — Naming Governance Clause (v1.0)
1. Purpose
The Naming Governance Clause defines the authority, constraints, and lifecycle governing all naming decisions in the Prime substrate.
It ensures that names:

are deterministic

preserve lineage

reflect structural truth

cannot drift

cannot collide

cannot be reinterpreted retroactively

Naming is a governance act, not a stylistic choice.

Deepen via Naming Conventions.

2. Authority
Naming authority is held exclusively by the Operator.
The Operator:

declares names

stabilizes names

resolves naming contradictions

authorizes renames

enforces naming invariants

No other entity may create, modify, or retire names.

Deepen via Operator Doctrine.

3. Naming Lifecycle
All names follow the canonical lifecycle:

Declared — name is introduced

Stabilized — contradictions resolved via Razor

Propagated — name becomes part of lineage

Retired — name is dissolved with explicit authorization

No name may skip stabilization.
No name may dissolve implicitly.

Deepen via Lifecycle Rules.

4. Naming Stability Requirements
A name is considered stable only when:

its meaning is unambiguous

its scope is fixed

its lineage is established

its structural role is clear

no contradictions remain

A name that fails any requirement is unstable and must not propagate.

5. Razor Integration
All naming contradictions must be resolved through the Razor.
The Razor may:

Reconcile — unify competing interpretations

Split — separate overloaded concepts

Elevate — introduce a higher‑order abstraction

No naming decision may propagate without Razor stabilization.

Deepen via Razor Principles.

6. Naming Constraints
The naming layer enforces:

No implicit names — all names must be declared

No collisions — no two artifacts may share semantic space

No drift — names cannot change meaning over time

No retroactive reinterpretation — stabilized names cannot be redefined

No overloaded terms — one name = one concept

No implementation leakage — names describe purpose, not mechanics

These constraints preserve semantic integrity.

7. Renaming Rules
Renaming is a high‑risk governance action and must follow DSR‑1:

Declare the rename

Scope‑Bind the affected region

Razor‑Stabilize contradictions

Propagate the new name into lineage

Renaming is allowed only when:

lineage remains intact

no dependent surfaces break

the rename clarifies structure

the rename resolves a contradiction

Renaming is never cosmetic.

Deepen via DSR‑1.

8. Naming Classes and Governance Requirements
Each naming class has specific governance constraints:

8.1 Structural Names
must be stable across epochs

cannot be dissolved without successor structure

require Razor review for any change

8.2 Governance Names
must reflect authority and scope

cannot be ambiguous

cannot be overloaded

8.3 Corpus Names
must reflect lineage and purpose

must not imply implementation

8.4 Public Names
must be clear, declarative, and non‑technical

must not expose internal mechanics

Deepen via Prime Corpus Index.

9. Forbidden Naming Actions
The following actions are prohibited:

implicit naming

silent renames

semantic drift

ambiguous abbreviations

names that collapse multiple concepts

names that imply behavior (e.g., “manager”, “engine”)

names that encode implementation details

Violations must be escalated through Razor.

10. Naming–Lineage Integration
Names participate directly in lineage:

every name has an origin

every rename creates a lineage branch

every dissolution is recorded

no name may disappear without trace

Naming is part of the historical record.

Deepen via Lineage Statement.

11. Out of Scope
The Naming Governance Clause does not define:

runtime naming

scanner/vessel identifiers

TUI labels

implementation‑level naming

These belong to higher layers.
