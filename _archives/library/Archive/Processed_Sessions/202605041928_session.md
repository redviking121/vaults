Commenting inside Markdown is one of those things where **there is no official standard**, but there _are_ accepted practices depending on the toolchain. In Obsidian specifically, there are **three safe, non‑intrusive methods** — each with different behavior.

Here’s the clean, structural breakdown.

---

# **1. HTML Comments (Most Common, Obsidian‑Safe)**

This is the de facto standard across Markdown systems:

```
<!-- This is a comment -->
```

### Properties:

- Invisible in Preview
- Invisible in Live Preview
- Visible in Source Mode
- Does **not** affect rendering
- Supported everywhere Obsidian runs

### Use case:

- Internal notes
- Hidden metadata
- Explanations you don’t want rendered

This is the **recommended** method.

---

# **2. Obsidian Callout Comments (Visible but Stylized)**

If you want a comment that _is_ visible but clearly marked as non‑content:

```
> [!comment]
> This is a comment block.
```

### Properties:

- Visible
- Styled
- Non‑semantic
- Doesn’t interfere with your content structure

### Use case:

- Editorial notes
- TODOs
- Review comments
- Drafting annotations

This is good when you want to _see_ the comment but keep it separate from the main content.

---

# **3. YAML Comments (Frontmatter Only)**

Inside YAML frontmatter, you can use `#`:

```
---
title: Example
tags: [sample]
# This is a YAML comment
created: 2026-04-14
---
```

### Properties:

- Only works inside the frontmatter block
- Invisible to Obsidian’s renderer
- Useful for metadata notes

### Use case:

- Internal metadata
- Template hints
- Non-rendered configuration notes

---

# **4. What _not_ to do**

These patterns are **not** recommended:

### ❌ Using Markdown headings as comments

They pollute the outline.

### ❌ Using hidden text tricks (zero‑width spaces, etc.)

They break search and indexing.

### ❌ Using HTML tags like `<span style="display:none">`

Obsidian strips or sanitizes them.

---

# **5. Community Consensus**

Across Obsidian, GitHub, MkDocs, Hugo, and most Markdown ecosystems:

- **HTML comments** are the accepted standard
- **Callouts** are the Obsidian‑specific enhancement
- **YAML comments** are metadata‑only

This aligns with your goal of balancing community norms with your own workflow.
