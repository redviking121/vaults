# 🔥 **THE SSH DOCTRINE (the thing before the thing)**

This is the operator‑grade foundation that makes the rest of the lab predictable under fire.

## **1. Every machine is a sovereign identity**

No shared keys.  
No “copy the .ssh folder.”  
No shortcuts.

Each machine generates:

- its own `ed25519` keypair
- its own trust relationship with GitHub
- its own known_hosts fingerprint

This is non‑negotiable.  
This is how you prevent cross‑contamination and ghost failures.

---

## **2. The operator never assumes the agent is loaded**

Every machine must be able to answer one question cleanly:

```
ssh -T git@github.com
```

If that line doesn’t return:

```
Hi <user>! You've successfully authenticated...
```

— the machine is not allowed to touch the pipeline.

This is the “no exceptions under fire” rule.

---

## **3. The lab uses a single _pattern_, not a single _image_**

We don’t clone disks.  
We don’t clone home directories.  
We clone **process**.

The pattern is:

1. Generate key
2. Register key
3. Verify handshake
4. Clone repos
5. Build environment
6. Integrate with vault

This is how six machines behave like one organism.

---

## **4. No machine touches the real vault until it passes validation**

This is the safety interlock.

A machine must:

- authenticate
- clone
- build
- run Logseq
- export test data
- pass consistency checks

**Only then** does it get access to the real vault.

This prevents the “silent corruption” class of failures.

---

## **5. Every step is logged in the Studio Session**

Because this is living documentation, not a static manual.

We capture:

- the reasoning
- the decisions
- the corrections
- the operator logic

This is how future operators (or future you) understand _why_ the system works, not just _how_.
