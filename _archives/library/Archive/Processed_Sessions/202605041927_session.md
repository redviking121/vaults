## 1. System Identity (What Each Ecosystem _Is_)

### **Emacs**

- A **Lisp machine disguised as a text editor**.
- Fully programmable environment where _everything is code_ and _everything is extensible_.
- Functions, buffers, modes, UI, workflows—every layer is modifiable.

### **Obsidian**

- A **local‑first Markdown knowledge environment**.
- Core identity: _file‑based PKM_, _graph‑structured linking_, _plugin‑augmented workflows_.
- Extensible, but **not programmable at the substrate level**.

**Contrast:**  
Emacs is a programmable substrate; Obsidian is a structured application with plugin hooks.

---

## 2. Storage Model

### **Emacs**

- Operates on **buffers**, not files.
- Files are ephemeral sources/sinks; the _real_ object is the buffer.
- Arbitrary data structures can be stored, manipulated, and rendered.

### **Obsidian**

- Operates directly on **Markdown files**.
- The file is the canonical object; the editor is a view over the file.
- Metadata lives in YAML frontmatter or inline syntax.

**Contrast:**  
Emacs abstracts away the file; Obsidian centers the file.

---

## 3. Extensibility Architecture

### **Emacs**

- **Emacs Lisp** is the extension language.
- Extensions can modify _any_ part of the system.
- Modes (major/minor) define behavior, keymaps, syntax, UI, and workflows.
- No sandboxing; full system mutation is normal.

### **Obsidian**

- Extensions are **TypeScript plugins** running inside a constrained API.
- Plugins can:
    - read/write files
    - add views
    - add commands
    - extend UI panels
- Plugins **cannot** rewrite the core application or redefine the execution model.

**Contrast:**  
Emacs plugins are peers to the core; Obsidian plugins are guests of the core.

---

## 4. UI and Interaction Model

### **Emacs**

- UI is **text‑centric**, modal, keyboard‑driven.
- Windows/frames are abstractions over buffer views.
- UI is programmable; you can redefine the entire interface.

### **Obsidian**

- UI is **pane‑centric**, mouse‑friendly, plugin‑augmented.
- Panes correspond to files, searches, graphs, canvases.
- UI is _not_ fully programmable; plugins can add components but not redefine the UI paradigm.

**Contrast:**  
Emacs UI is code; Obsidian UI is a structured layout with extension points.

---

## 5. Linking and Knowledge Graph Semantics

### **Emacs**

- Linking is not native; must be implemented via packages (Org‑mode, Hyperbole, etc.).
- Graph semantics depend on the chosen package.
- No canonical graph model.

### **Obsidian**

- Linking is **first‑class**: `[[wikilinks]]`, backlinks, graph view.
- Graph is a core primitive with consistent semantics.
- Plugins extend but do not redefine the graph model.

**Contrast:**  
Obsidian has a built‑in graph ontology; Emacs has multiple competing ontologies.

---

## 6. Querying and Computation

### **Emacs**

- Computation is **native**: Emacs Lisp can compute anything.
- Querying depends on the package (Org‑mode tables, Org‑roam, etc.).
- Arbitrary computation can be embedded in documents.

### **Obsidian**

- Querying is plugin‑driven (Dataview, Datacore).
- Computation is sandboxed and limited to plugin APIs.
- No native computation model inside Markdown.

**Contrast:**  
Emacs is a computational environment; Obsidian is a document environment with computational add‑ons.

---

## 7. Plugin Ecosystem Dynamics

### **Emacs**

- Mature, decades‑old ecosystem.
- Packages often redefine entire workflows (Org‑mode, Magit).
- High fragmentation; multiple competing solutions for the same domain.
- Stability varies; some packages are effectively full subsystems.

### **Obsidian**

- Younger ecosystem with rapid growth.
- Plugins tend to be **modular augmentations**, not full subsystem replacements.
- Stronger consistency due to API constraints.
- Less fragmentation; more convergence around common patterns.

**Contrast:**  
Emacs plugins create new worlds; Obsidian plugins extend the existing world.

---

## 8. Philosophy and Design Constraints

### **Emacs**

- Philosophy: **Everything is text; everything is code; everything is mutable**.
- No hard boundaries between system and user.
- Infinite extensibility, infinite complexity.

### **Obsidian**

- Philosophy: **Local files, simple Markdown, extensible but stable**.
- Hard boundaries between core and plugins.
- Prioritizes stability, UX, and non‑technical users.

**Contrast:**  
Emacs is a Lisp machine; Obsidian is a knowledge tool.

---

## 9. Ecosystem Trajectory

### **Emacs**

- Slow evolution; stable substrate.
- Power users maintain long‑lived personal ecosystems.
- Innovation happens through packages, not core changes.

### **Obsidian**

- Rapid evolution; frequent core updates.
- Plugin ecosystem is expanding into:
    - databases
    - canvases
    - task systems
    - AI integrations
- Core team actively shapes the ecosystem direction.

**Contrast:**  
Emacs evolves horizontally; Obsidian evolves vertically.
