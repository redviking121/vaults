## **Structural Note Inspector — Operator Architecture v1.0**

### **1. Purpose (One Sentence)**

Expose the _structural state_ of the active note — frontmatter, headings, tags, links, invariants — in a clean, operator‑grade panel that updates live and warns on malformed structure.

---

## **2. Command Surface (User‑Facing Entry Points)**

- **Command: “Inspect Current Note Structure”**  
    Opens the inspector panel (or focuses it if already open).
    
- **Command: “Refresh Structural Inspection”**  
    Forces a re‑scan of the active file.
    
- **Command: “Toggle Structural Inspector Panel”**  
    Show/hide the persistent side panel.
    

These three are enough to validate the environment and give you a reusable pattern for future plugins.

---

## **3. UI Surface (What the User Sees)**

### **Panel Layout (Left‑to‑Right Priority)**

#### **A. Frontmatter Block**

- Status: present / missing / malformed
- Parsed YAML key‑value pairs
- Highlight missing required keys (configurable later)

#### **B. Headings Block**

- Ordered list of headings with depth
- Highlight:
    - jumps in hierarchy (e.g., H1 → H3)
    - duplicate headings
    - empty headings

#### **C. Tags Block**

- All tags found in frontmatter and body
- Highlight:
    - orphan tags (body only)
    - mismatched namespace patterns
    - forbidden tags (e.g., `xi:*` on decomposable notes)

#### **D. Links Block**

- Internal links
- External links
- Broken links (via metadata cache)
- Backlink count (optional)

#### **E. Structural Invariants Block**

Operator‑grade checks:

- Frontmatter must be first
- No content before frontmatter
- No empty sections
- No trailing whitespace on heading lines
- Required fields present (later configurable)

Each block is collapsible.  
Each block has a small icon indicating status: ✓ / ! / ✖.

---

## **4. Data Flow (How the Plugin Thinks)**

### **Trigger → Extract → Analyze → Render**

1. **Trigger**
    
    - Active file change
    - File modified
    - Manual refresh command
    - Panel opened
2. **Extract**
    
    - Read file contents
    - Parse frontmatter via Obsidian’s metadata cache
    - Parse headings via Markdown parser
    - Parse tags and links via metadata cache + regex fallback
3. **Analyze**
    
    - Run structural checks
    - Build a normalized “StructuralState” object:
        
        ```
        {
          frontmatter: {...},
          headings: [...],
          tags: [...],
          links: {...},
          invariants: [...]
        }
        ```
        
4. **Render**
    
    - Push StructuralState into the panel’s view
    - Apply status icons
    - Apply warnings
    - Apply color coding (green/yellow/red)

This is the reusable pattern for all future operator plugins:  
**Trigger → Extract → Analyze → Render.**

- Manual refresh command

Minimal but complete.

---

## **6. Error Modes (Operator‑Visible Failure States)**

- **No active file**  
    Panel shows: “No note selected.”
    
- **Unsupported file type**  
    Panel shows: “Not a Markdown file.”
    
- **Malformed frontmatter**  
    Panel shows YAML parse error with line number.
    
- **Metadata cache unavailable**  
    Panel shows: “Metadata not ready — retrying…”
    
- **Unexpected exception**  
    Panel shows: “Inspector error — see console.”  
    (Operator‑grade honesty, no silent failure.)

## **8. Operator Notes (The Philosophy)**

- **Shape over content.**  
    The inspector never interprets meaning — only structure.
    
- **Zero drift.**  
    The panel always reflects the current file state.
    
- **No magic.**  
    Every warning is explicit, deterministic, and explainable.
    
- **Reusable pattern.**  
    This plugin becomes the template for all future operator tools:
    
    - ingest
    - normalize
    - analyze
    - surface

It’s the _instrument_ you build before you build the _weapons_.
