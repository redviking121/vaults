### TypeScript build pipeline

**Core objectives:** determinism, fast feedback, artifact traceability.

1. **Source layout**
    
    - **Label:** `apps/`, `packages/`, `tooling/`
    - **Pattern:**
        - **apps:** entrypoints (CLI, services, UI)
        - **packages:** shared libs (domain, infra, logging, schema)
        - **tooling:** build scripts, codegen, CI helpers
2. **Compilation stages**
    
    - **Stage 1 – Typecheck only:**
        - `tsc --noEmit` (strict mode, incremental, composite projects)
        - Fails fast; no emit on type errors.
    - **Stage 2 – Transpile:**
        - `tsup` / `esbuild` / `swc` for JS emit (CJS+ESM targets)
        - Sourcemaps mandatory; build metadata file per artifact.
    - **Stage 3 – Bundling (if needed):**
        - Separate configs for **server**, **browser**, **lambda/edge**.
        - Tree‑shaking + externals pinned (e.g., `node_modules`).
3. **Configuration**
    
    - **Root `tsconfig.base.json`:** strict, path aliases, module resolution.
    - **Per‑package `tsconfig.json`:** references, output dirs, `composite: true`.
    - **Build graph:** `tsc --build` or `turbo`/`nx` to respect dependency DAG.
4. **Quality gates**
    
    - **Lint:** `eslint` with TS plugin; run pre‑typecheck or in parallel.
    - **Format:** `prettier` as non‑blocking but enforced in CI.
    - **Tests:** `vitest`/`jest` with `--runInBand` for flaky suites in CI.
5. **CI pipeline**
    
    - **Step 1:** restore cache (node_modules, `.turbo`, `.tsbuildinfo`).
    - **Step 2:** `lint` + `tsc --noEmit`.
    - **Step 3:** build artifacts per app/package.
    - **Step 4:** tests (unit → integration → e2e).
    - **Step 5:** publish artifacts (npm, internal registry, container images).

### Operator‑grade logging subsystem

**Core objectives:** observability under stress, low cognitive load, structured by default.

1. **Log model**
    
    - **Shape:**
        - `timestamp`, `level`, `service`, `env`, `trace_id`, `span_id`, `op`, `msg`, `data`, `err`.
    - **Format:** JSON only for machine paths; human‑pretty only at dev TTY.
2. **Levels and semantics**
    
    - **TRACE:** noisy internals, disabled in prod by default.
    - **DEBUG:** operator‑useful state, not per‑request spam.
    - **INFO:** lifecycle, major decisions, ingest milestones.
    - **WARN:** degraded mode, retries, soft failures.
    - **ERROR:** request‑level failure, recoverable.
    - **FATAL:** process‑level failure, triggers restart/page.
3. **Context propagation**
    
    - **Per request:** `trace_id`, `span_id`, `user_id` (if applicable).
    - **Per component:** `service`, `version`, `build_id`, `node`.
    - Implement via request‑scoped context (CLS/AsyncLocalStorage) or explicit context passing.
4. **Implementation pattern**
    
    - **Core logger:** thin wrapper over `pino`/`winston`/custom, with:
        - Structured fields, redaction rules, level filters.
        - Sinks: stdout (primary), optional sidecar (UDP/TCP) if needed.
    - **Logger factory:** `createLogger(context)` returns bound logger with default fields.
    - **No naked console.log:** banned by lint rule.
5. **Pipelines and sinks**
    
    - **App → stdout → collector:**
        - Sidecar/agent (Fluent Bit, Vector, etc.) tails stdout.
        - Ships to central store (Loki, Elasticsearch, OpenSearch, etc.).
    - **Indexing:**
        - Index by `service`, `env`, `trace_id`, `op`, `level`.
        - Retention tiers: hot (hours–days), warm (weeks), cold (archive).
6. **Operator affordances**
    
    - **Canonical ops fields:** `op` (operation name), `phase`, `attempt`, `duration_ms`.
    - **Playbooks:** for each critical `op`, define:
        - What WARN/ERROR patterns look like.
        - First three commands/queries an operator runs.
    - **Sampling:**
        - Full logs for ERROR/FATAL.
        - Sampled DEBUG/TRACE in high‑traffic paths.

### Ingest pipeline architecture

**Core objectives:** backpressure, idempotency, replay, and observability.

1. **High‑level stages**
    
    - **Stage 0 – Edge:** accept input (HTTP, gRPC, file drop, message bus).
    - **Stage 1 – Normalize:** validate, enrich, canonicalize into internal schema.
    - **Stage 2 – Queue:** durable buffer (Kafka, NATS JetStream, SQS, etc.).
    - **Stage 3 – Process:** transform, route, fan‑out.
    - **Stage 4 – Persist:** write to OLTP/OLAP/object storage/search.
    - **Stage 5 – Index/Serve:** materialized views, search indices, caches.
2. **Contracts and schemas**
    
    - **External contracts:** versioned, documented; strict validation at edge.
    - **Internal schema:** stable canonical event shape; versioned with migrations.
    - **Idempotency:** keys derived from source (e.g., `(source_id, source_seq)`).
3. **Backpressure and reliability**
    
    - **Edge:**
        - Reject or 429 when queue lag exceeds threshold.
        - Rate limiting per tenant/source.
    - **Queue:**
        - At‑least‑once delivery; consumer groups per processing role.
        - Dead‑letter queues for poison messages with reason codes.
    - **Processors:**
        - Idempotent writes; retries with jitter; circuit breakers to downstreams.
4. **Storage strategy**
    
    - **Raw store:** immutable, append‑only (object storage) for replay/audit.
    - **Curated store:** cleaned/normalized tables (data warehouse / DB).
    - **Serving indices:** search (e.g., logs/metrics), aggregates, feature stores.
5. **Observability hooks**
    
    - **Metrics:**
        - Ingest rate, error rate, end‑to‑end latency, queue depth, DLQ volume.
    - **Logs:**
        - Log at stage boundaries with `op=ingest.stageX`, `source`, `tenant`, `batch_id`.
    - **Tracing:**
        - Single trace from edge to persistence; spans per stage.
6. **Operational controls**
    
    - **Kill switches:** per‑source and per‑stage toggles.
    - **Replay:** ability to re‑ingest from raw store or queue offsets.
    - **Config:** declarative routing rules (YAML/DB) rather than code edits.
