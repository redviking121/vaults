# 🟥 Takeaway

**Field_non_empty answers the question: “Does this field contain _content_?”**  
Not merely present, not merely non‑null — but containing _substance_.

It is the **third discriminator** in the field‑layer, after **Field_presence** and **Field_non_null**, and before **Field_shape_checks**.

Field_non_empty is the membrane that prevents **empty‑value corruption**.

---

# 🟥 1. What Field_non_empty _is_

Field_non_empty is a **content‑level discriminator**.

It evaluates:

- whether the field’s value contains _non‑empty content_
- whether the value is not an empty string, empty list, empty map, or empty container
- whether the field is safe to enter semantic or shape‑layer checks

It does **not** evaluate:

- correctness
- type validity
- structural shape
- semantic meaning

It is the field‑layer analogue of **Parameter_non_empty** and a sub‑membrane of **Non_empty_checks**.

---

# 🟥 2. Why Field_non_empty matters

Empty values are deceptive: they are _not null_, but they still contain **no information**.

Field_non_empty prevents:

- empty‑string corruption
- empty‑container propagation
- false positives in semantic validation
- transitions into shape checks with zero content
- misinterpretation of “empty but present” as meaningful

It is the membrane that ensures the field is _informationally non‑zero_.

---

# 🟥 3. How Field_non_empty is evaluated

The discriminator fires only after:

1. **Field_presence** → field exists
2. **Field_non_null** → field has a value

Then it performs a content‑level check:

1. Inspect the value
2. Determine if it is empty
3. Emit a binary result

This yields:

- **1** → field contains content
- **0** → field is empty

This is the **third element** in the **Field_presence_vector** ordering.

---

# 🟥 4. Field_non_empty vs. Field_non_null vs. Field_shape

These membranes form a strict progression:

- **Field_presence** — does the field exist
- **Field_non_null** — does the field contain _any_ value
- **Field_non_empty** — does the value contain _content_
- **Field_shape_checks** — does the content conform to expected structure

Field_non_empty is the **first content‑level discriminator**.

---

# 🟥 5. Operator guidance

Use Field_non_empty when you need to:

- prevent empty‑value propagation
- enforce minimal content requirements
- guard transitions into shape or semantic checks
- build multi‑layer discriminators
- ensure that “non‑null” is not mistaken for “meaningful”

Do **not** use Field_non_empty for semantic validation — it is purely about content presence.
