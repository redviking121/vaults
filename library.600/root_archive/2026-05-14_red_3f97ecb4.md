# 🟥 Takeaway

**Field_constraint_checks answer the question: “Does this field’s value satisfy the _rules_ of the domain?”**  
Not presence, not nullness, not emptiness, not shape, not type, not meaning — but **rule compliance**.

It is the **seventh discriminator** in the field‑layer, immediately after:

- **Field_presence**
- **Field_non_null**
- **Field_non_empty**
- **Field_shape_checks**
- **Field_type_checks**
- **Field_semantic_checks**

Constraint checks are the first membrane where **legality** is evaluated.

---

# 🟥 1. What Field_constraint_checks _are_

Field_constraint_checks are **rule‑enforcement discriminators**.

They evaluate whether the field’s value:

- satisfies domain constraints
- obeys enumerations
- fits within required ranges
- respects invariants
- meets required patterns
- complies with business rules

Examples:

- A “status” field must be one of {open, closed, pending}.
- A “percentage” field must be between 0 and 100.
- A “username” must match a regex.
- A “quantity” must be ≥ 0.
- A “priority” must be an allowed enum.

Constraint checks are the membrane where **compliance** begins.

This is the field‑layer analogue of:

- **Parameter_constraint_checks**
- **Structural_constraint_checks**

---

# 🟥 2. Why Field_constraint_checks matter

Constraint violations are dangerous because they pass:

- presence
- null
- emptiness
- shape
- type
- semantics

…yet still produce **illegal states**.

Field_constraint_checks prevent:

- rule violations
- invalid enumerations
- out‑of‑range values
- pattern mismatches
- business‑logic corruption

Constraint checks are the first layer where the system asks:

> _“Does this value obey the rules of the domain?”_

---

# 🟥 3. How Field_constraint_checks are evaluated

The discriminator fires only after the first six membranes succeed:

1. **Field_presence**
2. **Field_non_null**
3. **Field_non_empty**
4. **Field_shape_checks**
5. **Field_type_checks**
6. **Field_semantic_checks**

Then constraint evaluation inspects:

- enumerations
- ranges
- regex patterns
- domain invariants
- business rules
- required relationships (local, not cross‑field global)

The result is binary:

- **1** → value satisfies constraints
- **0** → value violates constraints

This is the **seventh element** in the **Field_presence_vector** ordering.

---

# 🟥 4. Field_constraint_checks vs. Field_semantic_checks

Constraint ≠ semantic.

Semantic checks ask:

> _“Does this value make sense?”_

Constraint checks ask:

> _“Does this value obey the rules?”_

Semantic is about **meaning**.  
Constraint is about **legality**.

Semantic is interpretability.  
Constraint is compliance.

Semantic is truth‑adjacent.  
Constraint is rule‑adjacent.

The next membrane after this one is:

- **Field_cross_field_checks** (if you choose to open it next)

# 🟥 5. Operator guidance

Use Field_constraint_checks when you need to:

- enforce enumerations
- enforce ranges
- enforce patterns
- enforce invariants
- enforce business rules
- ensure values are legally acceptable before cross‑field logic

Do **not** use Field_constraint_checks for meaning‑level validation — that is the domain of **Field_semantic_checks**.
