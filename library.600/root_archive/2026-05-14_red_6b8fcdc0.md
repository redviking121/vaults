# 🟥 1. What Parameter_cross_parameter_checks _are_

They validate **relationships between parameters**, not the parameters themselves.

A parameter may be:

- present
- non‑null
- non‑empty
- shaped correctly
- typed correctly
- semantically meaningful
- constraint‑compliant

…and still be **invalid in relation to another parameter**.

Cross‑parameter checks enforce:

- relational invariants
- dependent constraints
- conditional rules
- mutual exclusivity
- ordering
- consistency across parameters

Examples:

- `limit ≥ offset`
- `min_age ≤ max_age`
- `auth_method = "oauth"` requires `token`
- `mode = "manual"` forbids `auto_config`
- `currency` must match `region`
- `page_size` must be compatible with `pagination_mode`

This is the membrane where **local parameter correctness** becomes **relational correctness**.

# 🟥 2. Why Parameter_cross_parameter_checks matter

Without cross‑parameter checks, an invocation can pass all earlier membranes yet still produce:

- contradictory invocation states
- illegal parameter combinations
- impossible configurations
- broken invariants
- relational corruption

Cross‑parameter checks prevent:

- mismatched ranges
- incompatible modes
- missing dependent parameters
- illegal combinations
- contradictory flags

This is the parameter‑layer analogue of:

- **Field_cross_field_checks**
- **Structural_cross_component_checks**

# 🟥 3. How Parameter_cross_parameter_checks are evaluated

The discriminator fires **only after** each parameter is individually validated.

Then it evaluates:

- **pairwise relationships** (A ↔ B)
- **multi‑parameter invariants** (A, B, C must satisfy X)
- **conditional dependencies** (if A then B)
- **mutual exclusivity** (A xor B)
- **ordering constraints** (A < B < C)
- **coherence rules** (A implies B’s domain)

The result is binary:

- **1** → relationally valid
- **0** → relationally invalid

This is the **eighth element** in the **Parameter_presence_vector** ordering.

# 🟥 4. Cross‑parameter vs. Constraint vs. Semantic

These three membranes are often conflated. Their boundaries:

- **Parameter_semantic_checks**  
    “Does this value _make sense_ on its own?”
    
- **Parameter_constraint_checks**  
    “Does this value obey the _rules_ of the domain?”
    
- **Parameter_cross_parameter_checks**  
    “Does this value remain valid _in relation to other parameters_?”
    

Semantic = meaning.  
Constraint = legality.  
Cross‑parameter = relational coherence.

---

# 🟥 5. Operator guidance

Use Parameter_cross_parameter_checks when:

- a parameter’s validity depends on another parameter
- relational invariants must be enforced
- dependent parameters must be present
- ordering or ranges must be respected
- combinations must be legal
- invocation‑level rules span multiple parameters

Do **not** use Parameter_cross_parameter_checks for:

- single‑parameter semantics
- single‑parameter constraints
- structural or type validation

This membrane is the bridge between **parameter‑local correctness** and **invocation‑level correctness**.
