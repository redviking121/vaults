# 🟥 Takeaway

**inventory_version is the _temporal identity_ of the entire Inventory snapshot — the version of the _Inventory system itself_, not the schema of any individual entity.**  
Where **schema_version** describes _shape_,  
**inventory_version describes _epoch_.**

It tells the Engine:  
_“This snapshot was produced by Inventory Engine vX.Y.Z — interpret all semantics accordingly.”_

This is the version anchor for the **producer**, not the **entity**.

---

# 🟥 1. What inventory_version _is_

inventory_version is a **semantic version** attached to the _Inventory capture as a whole_, typically at the root of the snapshot:

```
{
  "inventory_version": "0.4.1",
  "host": "...",
  "entities": [...]
}
```

It binds the snapshot to the **Inventory Engine contract** that produced it.

This is distinct from:

- **schema_version** — version of an entity’s shape
- **format_field** — version of the decoding grammar
- **Explicit_version_field** — strongest per‑entity version anchor

inventory_version is the **global** version anchor.

---

# 🟥 2. Why the Diff Tool needs inventory_version

The Diff Tool uses inventory_version to:

- interpret global semantics (naming rules, entity categories, canonical fields)
- resolve differences in Inventory Engine behavior across versions
- apply the correct **Normalize_schemas** rules
- detect incompatible snapshots (e.g., v0.3 vs v0.6)
- ensure reproducibility of diffs across time

Without inventory_version, the Engine would have to infer the producer epoch — a source of drift.

With it, the Engine knows exactly which **Inventory contract** to apply.

---

# 🟥 3. The three roles of inventory_version

inventory_version participates in three membranes:

- **Version_signal** — global discriminator
- **Schema_detection** — producer‑epoch inference
- **Normalize_schemas** — epoch‑aware canonicalization

It is the _first_ signal the Engine reads when loading a snapshot.

# 🟥 4. Operator rules for inventory_version

To maintain substrate stability:

- increment **MAJOR** when global semantics change
- increment **MINOR** when new entity types or fields are added
- increment **PATCH** for non‑breaking clarifications
- never rewrite history — snapshots must remain tied to their original producer version
- always include inventory_version at the root of every snapshot

This ensures the entire Inventory substrate remains drift‑proof across time.
