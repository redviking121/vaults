# **Axiom Linter Specification (Generic Template)**

**Version:** 0.1 (Template)  
**Status:** Draft  
**Purpose:** Provide a universal, implementation‑agnostic specification for a linter that validates “axioms” — foundational, atomic truths — within any knowledge system, codebase, or document repository.

---

## **1. Overview**

An **Axiom Linter** is a validation engine that ensures all axioms in a system:

- follow a consistent structure
- adhere to naming and versioning rules
- maintain internal logical integrity
- maintain cross‑axiom coherence
- remain stable, atomic, and non‑contradictory

This spec defines the **contract**, **rule model**, **error taxonomy**, and **extensibility framework** for any implementation.

---

## **2. Definitions**

- **Axiom:** A short, atomic, stable statement that expresses a foundational truth within a system.
- **Axiom File:** A document containing exactly one axiom and its metadata.
- **Axiom Set:** The complete collection of axioms under linting.
- **Violation:** Any deviation from the rules defined in this spec.
- **Linter:** The engine that evaluates axioms and emits structured results.

---

## **3. Axiom File Structure Requirements**

Each axiom file MUST contain:

1. **Identifier**
    
    - Format: `ax-<domain>-<number>`
    - Example: `ax-ops-001`
    - MUST be globally unique.
2. **Title**
    
    - Short, declarative, ≤ 80 characters.
3. **Statement**
    
    - One atomic sentence.
    - MUST NOT contain conjunctions that imply multiple truths (“and”, “or”, “but”) unless part of a fixed phrase.
4. **Rationale**
    
    - 1–3 sentences explaining why the axiom exists.
5. **Implications**
    
    - Bullet list of consequences or constraints derived from the axiom.
6. **Version Metadata**
    
    - `version:` semantic version
    - `last_updated:` ISO‑8601 date
    - `status:` draft | stable | deprecated
7. **Optional Fields**
    
    - `tags:`
    - `links:` (references to other axioms or documents)

---

## **4. Directory Layout Requirements**

The linter assumes a canonical structure:

```
/axioms
    /<domain>/
        ax-<domain>-<number>.md
    /00-meta/
        000-index.md
```

Rules:

- Each domain MUST have its own directory.
- Each file MUST contain exactly one axiom.
- Index files MUST NOT contain axioms.

---

## **5. Rule Model**

The linter enforces three classes of rules:

### **5.1 Structural Rules**

Validate the shape of each axiom file.

Examples:

- Identifier format is correct.
- Required fields exist.
- Statement is atomic.
- Version metadata is present and valid.

### **5.2 Semantic Rules**

Validate meaning, coherence, and logical integrity.

Examples:

- No axiom contradicts another.
- No axiom duplicates the meaning of another.
- Implications do not introduce new axioms.
- Rationale aligns with the statement.

### **5.3 Referential Rules**

Validate cross‑file relationships.

Examples:

- Links reference valid axiom IDs.
- No circular dependency chains.
- Deprecated axioms are not referenced by stable ones.

---

## **6. Linter Output Specification**

The linter MUST emit a structured result object:

```
{
  "summary": {
    "files_scanned": <int>,
    "violations": <int>,
    "errors": <int>,
    "warnings": <int>
  },
  "results": [
    {
      "file": "axioms/ops/ax-ops-001.md",
      "status": "ok" | "warning" | "error",
      "messages": [
        {
          "rule_id": "AX-STRUCT-001",
          "severity": "error",
          "message": "Identifier does not match required pattern.",
          "line": 3
        }
      ]
    }
  ]
}
```

Requirements:

- MUST be JSON‑serializable.
- MUST be deterministic.
- MUST include machine‑readable rule IDs.

---

## **7. Error Taxonomy**

### **7.1 Severity Levels**

- **Error:** Violates a mandatory rule.
- **Warning:** Violates a recommended rule.
- **Info:** Non‑blocking observation.

### **7.2 Error Categories**

- `AX-STRUCT-*` — structural violations
- `AX-SEM-*` — semantic violations
- `AX-REF-*` — referential violations
- `AX-META-*` — metadata violations
- `AX-SYS-*` — system or parsing errors

---

## **8. Configuration Model**

The linter MUST support a configuration file:

```
axiom-linter.config.json
```

Supported fields:

- `domains`: whitelist or blacklist
- `rules`: enable/disable specific rules
- `severity_overrides`: promote/demote rule severity
- `ignore`: list of files or directories
- `output_format`: json | text | table

---

## **9. Extensibility Model**

The linter MUST support:

- **Custom rules**
- **Custom domains**
- **Custom metadata fields**
- **Plugin modules** (language‑specific or domain‑specific)

Each extension MUST declare:

- rule ID prefix
- description
- severity
- evaluation function signature
- test cases

---

## **10. CLI Contract (Generic)**

A generic CLI SHOULD support:

```
axiom-lint <path> [options]
```

Options:

- `--config <file>`
- `--format json|text|table`
- `--fail-on-warning`
- `--domains <list>`
- `--rules <list>`
- `--verbose`
- `--fix` (if auto‑fixing is supported)

Exit codes:

- `0` — no errors
- `1` — errors found
- `2` — system failure

---

## **11. Test Suite Requirements**

A complete implementation MUST include:

- **Unit tests** for each rule
- **Fixture directories** for valid and invalid axiom sets
- **Golden output tests** for CLI
- **Regression tests** for semantic rules

---

## **12. Minimal Example Axiom (Template)**

```
id: ax-core-001
title: Truths must be atomic.
statement: An axiom must express exactly one foundational truth.
rationale: Atomicity ensures axioms remain stable, testable, and non‑ambiguous.
implications:
  - Multi‑truth statements must be decomposed.
  - Axioms cannot contain logical conjunctions that imply multiple truths.
version: 1.0.0
last_updated: 2026-01-01
status: stable
```

---

## **13. Completion Criteria**

An Axiom Linter implementation is considered complete when:

- All rules in this spec are implemented.
- The CLI contract is satisfied.
- The output format is stable.
- The test suite covers all rule categories.
- The linter can run against a real axiom set without manual intervention.
