# **Linter CLI Contract (v1.0)**

**Namespace:** `LNC-*`  
**Status:** Authoritative  
**Owner:** Red Horne  
**Scope:** Execution interface for AQL queries and AEP enforcement actions  
**Purpose:** Define the **command‑line contract** for interacting with the linter engine.

---

## **LNC‑01 — CLI Object Model**

The CLI exposes three primitives:

- **query** — execute an AQL query
- **lint** — run LN‑series rule checks
- **enforce** — execute AEP actions

All commands operate over a **snapshot** of the Library.

---

## **LNC‑02 — Command Structure**

### **LNC‑02.1 — Base form**

```
linter <command> [options] [arguments]
```

Where:

- `<command>` ∈ {`query`, `lint`, `enforce`, `version`, `help`}
- `[options]` = flags modifying execution
- `[arguments]` = command‑specific payloads

---

## **LNC‑03 — Commands**

---

### **LNC‑03.1 — `query`**

**Purpose:** Execute an AQL query against the current snapshot.

**Form:**

```
linter query --file <path>  
linter query --expr "<AQL expression>"
```

**Options:**

- `--file <path>` — load AQL from file
- `--expr "<string>"` — inline AQL expression
- `--format <json|table|raw>` — output format
- `--limit <n>` — restrict result count
- `--snapshot <id>` — run against a specific snapshot

**Output:**  
AQL result set (deterministic, read‑only).

**Failure modes:**

- `AQL-ERR-01` — invalid syntax
- `AQL-ERR-02` — unknown field
- `AQL-ERR-03` — illegal structural predicate

---

### **LNC‑03.2 — `lint`**

**Purpose:** Execute LN‑series rule checks.

**Form:**

```
linter lint  
linter lint --rules <pattern>
linter lint --note <id>
```

**Options:**

- `--rules <pattern>` — glob or comma‑list (e.g., `LN-1*`, `LN-201`)
- `--note <id>` — restrict linting to a single object
- `--format <json|table|raw>`
- `--snapshot <id>`

**Output:**  
List of violations with:

- rule id
- object id
- location
- severity
- structural description

**Failure modes:**

- `LN-ERR-01` — unknown rule
- `LN-ERR-02` — invalid object reference

---

### **LNC‑03.3 — `enforce`**

**Purpose:** Execute AEP actions.

**Form:**

```
linter enforce --action <AEP-action> --target <id>
```

**Options:**

- `--action <AEP-* id>` — e.g., `AEP-DETECT`, `AEP-STABILIZE`, `AEP-REPAIR`, `AEP-ESCALATE`
- `--target <id>` — object or violation id
- `--dry-run` — show actions without applying
- `--snapshot <id>`

**Output:**  
AEP action report:

- action
- target
- pre‑state
- post‑state
- structural deltas

**Failure modes:**

- `AEP-ERR-01` — illegal action for target state
- `AEP-ERR-02` — stabilization failure
- `AEP-ERR-03` — unresolved conflict

---

### **LNC‑03.4 — `version`**

```
linter version
```

Outputs:

- CLI version
- AQL version
- AEP version
- LN catalog version

---

### **LNC‑03.5 — `help`**

```
linter help
linter help <command>
```

Outputs structural usage only.

## **LNC‑04 — Exit Codes**

|Code|Meaning|
|---|---|
|`0`|success|
|`1`|AQL error|
|`2`|LN violation(s) detected|
|`3`|AEP action failure|
|`4`|invalid command|
|`5`|internal engine error|

Exit codes are **stable invariants**.

## **LNC‑06 — Snapshot Semantics**

All commands operate on a **snapshot**:

- default = current working state
- explicit snapshots must be immutable
- AEP actions produce new snapshots

Snapshot IDs are stable, non‑semantic identifiers.

---

## **LNC‑07 — Logging**

Two channels:

- **stdout** — primary output
- **stderr** — errors, warnings, diagnostics

No mixed channels.

---

## **LNC‑08 — Determinism Guarantees**

- identical input → identical output
- no randomness
- no time‑dependent behavior
- no external network calls

---

## **LNC‑09 — Security Model**

- read‑only for `query`
- read‑only for `lint`
- controlled mutation for `enforce`
- no side effects outside snapshot directory
