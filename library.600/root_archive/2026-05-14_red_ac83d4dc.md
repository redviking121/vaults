# **Pre‑Flight Drift Check — v0.3**

### **Takeaway:**

v0.3 adds **lineage** (the ancestry of a conflict) and **override codes** (operator‑safe interventions).  
This is the first version that can _explain itself_.

---

## 🧬 **1. New Concept: Conflict Lineage**

Every conflict now carries a **lineage tag**:

```
<CLASS>:<ORIGIN>:<DEPTH>
```

Where:

- **CLASS** — one of the v0.2 conflict classes  
    (`DIVERGENCE`, `TOMBSTONE`, `STALE_WRITE`, `ENVIRONMENT`, `AMBIGUOUS`)
- **ORIGIN** — the first event that created the drift  
    (`LOCAL_EDIT`, `REMOTE_EDIT`, `DELETE_LOCAL`, `DELETE_REMOTE`, `CLOCK_SKEW`, `FS_FAULT`)
- **DEPTH** — how many times this conflict has reappeared or mutated  
    (integer, starts at 0)

**Purpose:**  
Lineage prevents “Groundhog Day conflicts.”  
If the same file keeps drifting, the system knows it’s not a new problem — it’s a **recurrence**.

---

## 🧩 **2. Conflict Object (v0.3)**

Every conflict is now a structured object:

```
Conflict {
    class: <enum>,
    origin: <enum>,
    depth: <int>,
    local_hash: <string>,
    remote_hash: <string>,
    tombstone: <bool>,
    last_resolved: <timestamp|null>
}
```

This is the minimal set that gives you:

- classification
- ancestry
- recurrence detection
- operator auditability
- deterministic resolution

---

## 🔧 **3. Override Codes (Operator‑Safe)**

Override codes are **atomic**, **typed**, and **never mutate lineage**.  
They sit _outside_ the conflict object — they don’t rewrite history.

### **Allowed override codes:**

```
O-FORCE-LOCAL      → trust local, overwrite remote
O-FORCE-REMOTE     → trust remote, overwrite local
O-ACK-TOMBSTONE    → accept deletion as authoritative
O-RETRY            → re-run drift check after environment stabilizes
O-IGNORE           → mark conflict as acknowledged but unresolved
```

### **Rules:**

1. **Overrides never change lineage.**  
    They only append a `resolution_event`:
    
    ```
    resolution_event = {
        code: <override>,
        timestamp: <now>,
        operator: <id>
    }
    ```
    
2. **Overrides are only allowed after classification.**  
    No blind overrides.
    
3. **Overrides cannot be chained.**  
    One override per conflict per cycle.
    
4. **Overrides do not reduce depth.**  
    Recurrence is preserved.

## 🔄 **4. v0.3 Decision Flow (compressed)**

```
START
  ↓
Collect signals (L.hash, R.hash, tombstones, env)
  ↓
Classify conflict (v0.2 rules)
  ↓
Attach lineage (origin, depth)
  ↓
IF no conflict → SAFE_TO_SYNC
  ↓
IF conflict → emit Conflict object
  ↓
Check for operator override
  ↓
IF override present → apply override resolution
  ↓
Record resolution_event
  ↓
END (no automatic sync)
```

**Key shift:**  
v0.3 **never auto-syncs** after a conflict.  
It always waits for operator intent or a clean state.
