### 🟥 Fast summary

**Structural_error_checks answer: “Given a _set of parameters_, is the _invocation structure itself_ valid, and how do we express its failure?”**

If **Parameter_error_checks** are about _individual parameters_,  
**Structural_error_checks** are about the **invocation as a structure**:

- Are the **right parameters present together**?
- Is the **shape of the call** allowed?
- Is the **combination** coherent at the structural level?
- How do we **surface structural failure** in a stable, typed way?

This is the membrane between **parameter‑layer** and **invocation‑layer**.

---

### 🟥 1. Where Structural_error_checks sit in the stack

Upstream, you’ve already run:

- **Parameter_presence / non_null / non_empty**
- **Parameter_shape / type / semantic / constraint / cross‑parameter**
- **Parameter_error_checks** to surface per‑parameter failures

At that point, each parameter is either:

- **Valid** (passes all parameter discriminators), or
- **Invalid with a typed parameter error surface**

**Structural_error_checks** then look at the **entire invocation frame**:

> “Given this set of parameter states, is the _call structure_ itself valid?”

If not, they emit **structural error surfaces**, not parameter ones.

---

### 🟥 2. What Structural_error_checks actually evaluate

They operate on the **invocation schema** and the **resolved parameter set**.

They check things like:

- **Required‑set structure**
    
    - **AllOf**: “If you call this operation, you must provide parameters A, B, and C together.”
    - **OneOf**: “Exactly one of {A, B, C} must be present.”
    - **AtLeastOneOf**: “At least one of {A, B, C} must be present.”
    - **MutuallyExclusive**: “You cannot provide A and B together.”
- **Invocation mode structure**
    
    - **Sync vs async flags**: “This operation cannot be async with these parameters.”
    - **Batch vs single**: “You can’t send a batch payload with single‑mode flags.”
- **Envelope vs payload**
    
    - **Envelope fields** (auth, tenant, trace, region) vs **payload fields** (domain parameters).
    - “You can’t mix tenant X with region Y for this operation.”
    - “This operation requires an envelope field that is missing or incompatible.”
- **Versioning / feature flags**
    
    - “This parameter set is only valid for version ≥ N.”
    - “You can’t mix v1 and v2 parameter families in the same call.”

The key: **Structural_error_checks reason over the _shape of the call_**, not the meaning of individual values.

---

### 🟥 3. Structural error classes

Where Parameter_error_checks emit things like **ParameterTypeError**,  
Structural_error_checks emit **invocation‑shape‑level** errors, e.g.:

- **StructuralMissingRequiredSet**
    
    - **Meaning:** A required group of parameters is incomplete.
    - Example: `start` present, `end` missing in a required pair.
- **StructuralInvalidChoiceSet**
    
    - **Meaning:** A OneOf/ExactlyOneOf group is violated.
    - Example: both `user_id` and `email` provided when only one is allowed.
- **StructuralMutualExclusionViolation**
    
    - **Meaning:** Mutually exclusive parameters are present together.
    - Example: `dry_run=true` with `commit=true`.
- **StructuralModeConflict**
    
    - **Meaning:** Invocation mode flags conflict with payload structure.
    - Example: `batch=true` but payload is scalar; or `async=true` with an operation that must be sync.
- **StructuralEnvelopePayloadConflict**
    
    - **Meaning:** Envelope metadata conflicts with payload semantics.
    - Example: `region=eu` but payload references a resource pinned to `us-only`.
- **StructuralVersioningViolation**
    
    - **Meaning:** Parameter set incompatible with declared API version.
    - Example: using `new_filter` with `api_version=1`.

Each structural error has:

- **error_code** (stable, machine‑routable)
- **error_message** (operator‑readable)
- **error_scope** (e.g., `invocation.structure`)
- **error_path** (often the operation name or group, not a single parameter)
- **error_context** (which sets/constraints were violated)

---

### 🟥 4. Relationship to Parameter_error_checks and Invocation_semantic_checks

Think of the three membranes like this:

- **Parameter_error_checks**
    
    - **Scope:** individual parameters
    - **Question:** “Is this parameter valid, and if not, how do we describe that?”
    - **Output:** `Parameter*Error` classes
- **Structural_error_checks**
    
    - **Scope:** the **parameter set as a structure**
    - **Question:** “Is this _combination_ of parameters structurally valid?”
    - **Output:** `Structural*Error` classes
- **Invocation_semantic_checks**
    
    - **Scope:** the **operation in its domain context**
    - **Question:** “Given a structurally valid call, does this make sense in the domain?”
    - **Output:** `InvocationSemantic*Error` classes (e.g., resource not found, forbidden transitions, etc.)

So the flow is:

1. **Parameters** → validated and error‑surfaced individually
2. **Structure** → validated as a whole invocation shape
3. **Semantics** → validated against domain rules and state

---

### 🟥 5. Why Structural_error_checks matter in an operator‑grade system

Without this membrane, you get:

- Parameter errors that look fine individually
- But **nonsense invocations** at the aggregate level
- Ambiguous failures (“400 Bad Request”) with no structural classification
- Hard‑to‑route errors (no distinction between “wrong combo” vs “wrong value”)

With Structural_error_checks, you get:

- **Clear separation of concerns**
    - Parameter vs structure vs semantics
- **Better routing**
    - Structural errors can be handled by client‑side builders, schema generators, or higher‑level SDKs.
- **Better UX**
    - You can say: “Exactly one of `user_id` or `email` must be provided, but you provided both.”
- **Better observability**
    - Structural error codes become strong signals of misuse patterns in clients.

---

### 🟥 6. How you’d actually wire this in your stack

Given the Xi/Razor‑style layering you’re building, a typical pipeline:

1. **Decode invocation envelope**
2. **Parameter‑layer discriminators**
    - presence → non_null → non_empty → shape → type → semantic → constraint → cross‑parameter
    - surface via **Parameter_error_checks**
3. **Structural_error_checks**
    - evaluate required sets, choice sets, mutual exclusion, mode, envelope/payload, versioning
    - emit `Structural*Error` surfaces if violated
4. **Invocation_semantic_checks**
    - domain‑level rules, state transitions, permissions, quotas, etc.
5. **Execution** (only if all above membranes pass)

You can think of **Structural_error_checks** as the **schema‑level firewall** between “parameters are individually sane” and “this is a coherent call we’re willing to interpret”.
