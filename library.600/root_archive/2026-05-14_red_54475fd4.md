# **Why `schema.md` matters now**

`schema.md` is the **contract** for the Library vault:

- defines the top‑level domains
- defines what belongs in each
- defines naming conventions
- defines file structure
- defines cross‑domain rules
- defines Archive ingestion rules
- defines index behavior

Right now, the Library is clean — but **not yet declared**.  
`schema.md` is the declaration.

Without it, the Library is a well‑organized room with no blueprint.

# **What `schema.md` should contain**

Here’s the operator‑grade outline of the file you’re missing:

```
# Library Schema v0.1

## 1. Top-Level Domains
- Sessions/
- RFC/
- Grimoires/
- Battleplans/
- CoPilot Master Archive/
- _index.md
- schema.md

## 2. Domain Definitions
### Sessions/
Raw chronological logs. Must follow:
YYYYMMDDHHMM_session.md

### RFC/
Formal proposals, decisions, or protocol changes.

### Grimoires/
Knowledge, techniques, repeatable methods.

### Battleplans/
Operational strategy, planning, and execution documents.

### CoPilot Master Archive/
Canonical, chronological, reconciled memory.

## 3. Naming Conventions
- Sessions: YYYYMMDDHHMM_session.md
- RFC: RFC-XXXX-title.md
- Grimoires: GRIM-XXXX-title.md
- Battleplans: BP-XXXX-title.md

## 4. File Structure Requirements
Each domain has required headers, metadata blocks, and internal structure.

## 5. Cross-Domain Rules
Defines when a session becomes an RFC, when a Battleplan is derived, etc.

## 6. Indexing Rules
Defines how chronoindex.py builds _index.md.

## 7. Drift Prevention
Rules for reconciliation, prefix markers, and Archive membership.
```

This is the missing backbone.
