# 🟥 Takeaway

**Schema_detection is the membrane where the Diff Tool figures out _what kind of thing_ it is looking at before it tries to normalize it.**  
This is the moment the Engine stops treating snapshots as raw JSON and starts treating them as **typed, versioned, contract‑bound entities**.

Without schema detection, normalization would be blind.  
With it, normalization becomes deterministic.

---

# 🟥 1. Schema_detection is a three‑signal inference membrane

The membrane always extracts three signals from each snapshot:

- **Version_signal** — what schema version does this snapshot claim or imply
- **Entity_signal** — what entity types are present (Operator, Environment, Package, Service, etc.)
- **Field_pattern_signal** — what field shapes and aliases indicate drift, legacy formats, or partial upgrades

These three signals form the **SchemaFingerprint**.

---

# 🟥 2. How Schema_detection actually works

The membrane runs a deterministic inference sequence:

1. **Check explicit version fields**
    
    - `schema_version`
    - `inventory_version`
    - `format_version`  
        If present, this is authoritative.
2. **If missing, infer from structure**
    
    - presence/absence of canonical fields
    - legacy field names
    - nested vs flat entity shapes
    - known drift patterns from earlier versions
3. **If ambiguous, fall back to pattern matching**
    
    - regex‑based field detection
    - entity clustering
    - structural heuristics

The result is a **SchemaFingerprint** object that tells the normalization membrane exactly what rules to apply.

---

# 🟥 3. Why Schema_detection is a membrane, not a lookup

Because snapshots in the real world drift.

Machines evolve.  
Tools change.  
Operators forget to update.  
Fields appear, disappear, mutate.

A static lookup would fail.  
A membrane **adapts**.

Schema_detection is the adaptive boundary that:

- absorbs drift
- recognizes legacy formats
- identifies partial upgrades
- preserves unknown fields
- and still produces a stable fingerprint

This is what makes the Diff Tool resilient.

# 🟥 5. The SchemaFingerprint object (operator‑grade)

A SchemaFingerprint always contains:

- `version` — explicit or inferred
- `entities` — list of detected entity types
- `field_patterns` — alias maps, drift signatures, legacy markers
- `confidence` — high/medium/low inference confidence
- `extraneous` — fields that don’t belong to any known schema

This object is passed directly into **Field_canonicalization**.
