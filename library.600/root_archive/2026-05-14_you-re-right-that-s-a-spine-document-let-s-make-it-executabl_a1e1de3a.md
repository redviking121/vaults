### Auto‑zip TUI script (creates, zips, and base64‑encodes)

```bash
#!/usr/bin/env bash
set -euo pipefail

NAME="inventory_new_vault"
ZIP="${NAME}.zip"

if [ -d "$NAME" ]; then
  echo "[xi] removing existing ./$NAME"
  rm -rf "$NAME"
fi

mkdir -p "$NAME"

cat > "$NAME/INVENTORY_A_NEW_VAULT.md" <<'EOF'
# INVENTORY A NEW VAULT
### Xi/Razor Substrate — Intake Protocol (Final Operator Document)

---

## 1. Identity
- Name: Inventory‑a‑New‑Vault Protocol
- Role: Establish the first‑touch sequence for onboarding a new vault
- Function: Produce a clean, normalized, drift‑proof initial state

---

## 2. Operator Stance
- Treat the vault as **unknown terrain**
- Assume **no structure**, **no guarantees**, **no trust**
- Maintain **read‑only posture** until normalization gates are passed
- Prioritize **capture**, then **classification**, then **anchoring**

---

## 3. Intake Sequence (Top‑Level)
1. Acquire Source  
2. Perform Raw Scan  
3. Normalize Structure  
4. Establish Anchors  
5. Generate Inventory Report  
6. Publish to Ops‑Vault  

Each step is a gate.  
No step is skipped.  
No gate is soft.

---

## 4. Step 1 — Acquire Source
- Identify the vault’s origin (local, cloud, export, archive)
- Verify integrity (hash, size, completeness)
- Freeze the source into a read‑only snapshot
- Record:
  - Source path
  - Timestamp
  - Hash
  - Operator ID

Output: `source_snapshot/`

---

## 5. Step 2 — Perform Raw Scan
Scan the snapshot without interpretation.

Capture:
- Directory tree
- File count
- File types
- Hidden files
- Orphaned files
- Metadata anomalies
- Non‑UTF8 content
- Oversized binaries
- Duplicate filenames

Output: `raw_scan.md`

---

## 6. Step 3 — Normalize Structure
Apply Xi/Razor normalization rules:

- Convert all paths to canonical lowercase‑dash format
- Remove illegal characters
- Resolve collisions deterministically
- Move non‑markdown assets into `/assets`
- Move logs, exports, and debris into `/_debris`
- Convert `.txt` → `.md` where appropriate
- Generate `_index.md` stubs for directories

Output: `normalized_vault/`

---

## 7. Step 4 — Establish Anchors
Anchors define the vault’s structural truth.

Create:
- `README.md` (vault identity)
- `_index.md` (root index)
- `/ops/` (operations layer)
- `/incoming/` (ingest layer)
- `/systems/` (architecture layer)
- `/notes/` (content layer)
- `/archive/` (cold storage)

Record anchor hashes.

Output: `anchors.md`

---

## 8. Step 5 — Generate Inventory Report
Produce a single operator‑grade report containing:

- Vault identity
- Source snapshot details
- Raw scan summary
- Normalization actions
- Anchor map
- File counts by type
- Directory topology
- Detected anomalies
- Operator notes

This becomes the canonical intake artifact.

Output: `inventory_report.md`

---

## 9. Step 6 — Publish to Ops‑Vault
Publish the following into the Ops‑Vault:

- `inventory_report.md`
- `anchors.md`
- `raw_scan.md`
- `normalized_vault/` (or link)
- Hash manifest

Placement:
- `Ops‑Vault/incoming/` for ingestion
- `Ops‑Vault/systems/intake/` for archival

Operator confirms:
- No partial syncs
- No missing anchors
- No drift from emitted structure

---

## 10. Completion Gate
A vault is considered onboarded when:

- All anchors resolve
- All hashes match
- No anomalies remain unclassified
- Inventory report is published
- Chronoindex has run at least once

Only then does the vault enter active governance.

---

## 11. Operator Notes (Blank Section)
Use this section to record:
- Deviations
- Observations
- Edge cases
- Structural decisions

This section is intentionally empty.
EOF

cat > "$NAME/_manifest.txt" <<'EOF'
inventory_new_vault/
  INVENTORY_A_NEW_VAULT.md
  _manifest.txt
EOF

[ -f "$ZIP" ] && rm -f "$ZIP"

echo "[xi] zipping -> $ZIP"
zip -qr "$ZIP" "$NAME"

echo "[xi] base64 of $ZIP (suitable for embedding):"
base64 "$ZIP"
```

Usage:

```bash
chmod +x emit_inventory_new_vault.sh
./emit_inventory_new_vault.sh > inventory_new_vault_zip.b64
```
