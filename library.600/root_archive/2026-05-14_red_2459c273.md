# 🟥 1. Version_signal is a three‑tier discriminator

The Engine extracts version identity using three independent but converging signals:

- **Explicit_version_field** — a declared `schema_version` or `version` field
- **Structural_signature** — the shape‑based fingerprint of the entity
- **Field_presence_vector** — the presence/absence pattern of canonical fields

These three signals form a **version inference lattice**.  
If one is missing, the others still converge.

---

# 🟥 2. Explicit_version_field (the strongest signal)

If the snapshot includes a direct version declaration, the membrane locks immediately:

- `schema_version: "1.0"`
- `inventory_version: 3`
- `format: "v2"`

This is the **hard anchor** — the Engine trusts it unless contradicted by impossible structure.

If explicit version exists, the other signals become **validators**, not deciders.

---

# 🟥 3. Structural_signature (shape‑based inference)

When no explicit version is present, the Engine computes a **structural signature**:

- field ordering
- nested object layout
- array vs object distinctions
- canonical field groupings

Each schema version has a unique structural fingerprint.  
This is the **soft anchor** — highly reliable, but still validated by the presence vector.

---

# 🟥 4. Field_presence_vector (the tie‑breaker)

This is the **binary vector** of which canonical fields exist:

- `["packages", "services", "kernel"]` → likely v1
- `["packages", "services", "kernel", "hardware"]` → likely v2
- `["packages", "services", "kernel", "hardware", "mounts"]` → likely v3

This vector resolves ambiguity when structural signatures overlap across versions.
