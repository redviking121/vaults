# 🟦 Takeaway

**Auxiliary capability is the membrane that declares _supportive, non‑critical system functions_ — capabilities that enhance or extend the entity but are _not required for its existence_.**  
It sits directly beneath **Core capability** and above **Specialized capability** in the **Capability role** stack.

If Core capability is _what the system cannot live without_,  
Auxiliary capability is **what the system benefits from**.

These are the “muscles and tools,” not the “organs.”

---

# 🟦 1. What Auxiliary capability actually encodes

Auxiliary capabilities are:

- supportive
- optional (in the existential sense)
- performance‑enhancing
- domain‑agnostic
- substrate‑adjacent

They improve the system’s usefulness, efficiency, or ergonomics, but the system can still operate without them.

Examples of Auxiliary capabilities:

- **logging** — observability, audit trails
- **caching** — performance acceleration
- **compression** — storage/network efficiency
- **metrics** — telemetry, monitoring
- **scheduling** — timed or deferred execution

These are not existential dependencies — they are _operational enhancers_.

# 🟦 2. The three capability strata (and where Auxiliary sits)

Auxiliary capability is the **middle layer** of the capability stack:

1. **Core capability** — foundational, mandatory
2. **Auxiliary capability** — supportive, optional
3. **Specialized capability** — domain‑specific, advanced

Auxiliary is the only tier that is:

- flexible
- swappable
- performance‑oriented
- context‑dependent

It is the “middleware” of the capability ontology.
